# Чеклист к собеседованию — что проверять сразу

Краткая шпаргалка по задачам из этого репозитория. Идея простая: до того как
писать код, прогнать правильные вопросы, чтобы не споткнуться о крайний случай и
не выбрать неподходящий паттерн.

> 🇬🇧 English version: [interview_checklist_en.md](interview_checklist_en.md)

---

## 0. До первой строчки кода (любая задача)

1. **Уточни вход.** Размер `n`? Диапазон значений? Отрицательные? Отсортирован?
   Возможны дубликаты? Может ли быть пустым?
2. **Проговори крайние случаи вслух:** пустой вход, один элемент, все элементы
   равны, минимум/максимум значений, переполнение int.
3. **Контракт вывода.** На месте (in-place) или вернуть новое? Индекс или
   значение? Важен ли порядок?
4. **Выбери сложность из ограничений** (см. шпаргалку внизу): размер `n` обычно
   подсказывает ожидаемую временную сложность.
5. **Сначала озвучь brute force, потом оптимизируй.** Сперва корректный базовый
   вариант, потом уже хитрый трюк.

## 1. Массивы (arrays)

**Сразу проверь:** отсортирован или нет? дубликаты разрешены? нужно ли in-place?
есть ли отрицательные?

**Основные паттерны:**
- Хеширование для поиска за O(1) → *Two Sum*
- Два указателя по отсортированному / in-place массиву → *Move Zeroes*,
  *Remove Duplicates*
- Префиксные суммы / бегущие итоги
- Скользящее окно для задач на подмассивы

**Типичные ошибки:** off-by-one на границах; изменение списка во время итерации
по нему; переполнение int на суммах/произведениях.

---

## 2. Строки (strings)

**Сразу проверь:** ASCII или Unicode? регистрозависимость? пробелы по краям?
пустая строка?

**Основные паттерны:**
- Счётчик частот (`dict` / `collections.Counter`) → *First Unique Character*,
  *Isomorphic Strings*, *Palindrome Permutation*
- Два указателя с обоих концов → *Valid Palindrome II*
- Стек → *Valid Parentheses*, *Simplify Path*

**Типичные ошибки:** переполнение в *Reverse Integer* (обрезай до ±(2³¹−1));
строки в Python неизменяемы, поэтому собирай `list` и в конце `''.join()`.

---

## 3. Связные списки (linked lists)

**Сразу проверь:** пустой список? один узел? возможен ли цикл?

**Основные паттерны:**
- **Фиктивная голова (dummy head)** упрощает вставку/удаление в начале
- Быстрый и медленный указатели → *Linked List Cycle*, поиск середины
- Разворот (половины) списка на месте → *Palindrome Linked List*

**Типичные ошибки:** потеря ссылки на `next` до переподвязки; обращение к
`node.next`, когда `node` равен `None`.

---

## 4. Деревья (trees)

**Сразу проверь:** пустое дерево? один узел? BST или обычное бинарное? важна ли
глубина рекурсии?

**Основные паттерны:**
- DFS-рекурсия → *Maximum Depth*, *Same Tree*
- **Границы диапазона** для валидации BST, а не только сравнение с родителем →
  *Validate BST*
- BFS через `collections.deque` для обхода по уровням / проверки зеркальности →
  *Symmetric Tree*
- Inorder-обход BST даёт отсортированные значения → *Inorder Traversal*

**Типичные ошибки:** валидация BST только сравнением узла с прямым родителем
(нужен унаследованный диапазон `(low, high)`); глубокая рекурсия переполняет стек
на вырожденных (skewed) деревьях.

---

## 5. Поиск и динамическое программирование

**Сразу проверь:** монотонен ли массив (или пространство ответов) → бинарный
поиск? есть ли перекрывающиеся подзадачи / оптимальная подструктура → DP?

**Основные паттерны:**
- Бинарный поиск по отсортированному массиву или по ответу → *Search Insert
  Position*, *Sqrt(x)*
- DP снизу вверх / мемоизация → *Climbing Stairs*, *Triangle*, *Pascal's Triangle*
- Поиск цикла через `set` уже виденных состояний → *Happy Number*

**Типичные ошибки:** ошибки в границах бинарного поиска (`lo <= hi` против
`lo < hi` и как сдвигаешь `mid`); забытые базовые случаи в DP.

---

## 6. SQL

**Сразу проверь:** могут ли столбцы быть NULL? дубликаты? какова **гранулярность**
каждой таблицы (одна строка на что)? какой тип джойна сохранит нужные строки?

**Основные паттерны:**
- Оконные функции (`ROW_NUMBER`, `RANK`, `FIRST_VALUE`) обычно **быстрее и чище**,
  чем self-join + `GROUP BY`, для задач «топ-N в группе»
- CTE (`WITH`) для разбивки запроса на читаемые шаги
- `LEFT JOIN` + `COALESCE`, чтобы сохранить несовпавшие строки и подставить
  значения по умолчанию
- Логика пересечения интервалов (`a.start < b.end AND a.end > b.start`) →
  одновременные задачи, расписания

**Типичные ошибки:** `NOT IN` с NULL в подзапросе возвращает пусто (используй
`NOT EXISTS`); каждый неагрегированный столбец должен быть в `GROUP BY`; фильтр в
`WHERE` по правой таблице незаметно превращает `LEFT JOIN` в inner join.

---

## 7. Advent of Code (парсинг и симуляция)

**Сразу проверь:** точный формат входа? границы сетки? индексация с 0 или с 1?

**Основные паттерны:**
- **Парсинг → симуляция**: сначала чистые структуры данных, потом правила
- Представляй сетку как `dict` с ключом `(row, col)` для разреженных /
  безграничных сеток
- BFS / DFS для кратчайших путей и достижимости

**Конвенция репозитория:** каждый AoC-скрипт читает вход из `sys.argv[1]`, по
умолчанию `input.txt`. Запуск: `python aoc2024/aoc241.py my_input.txt`.

## Шпаргалка по сложности (размер `n` → допустимый подход)

| `n` до | Допустимая сложность | Типичная техника |
|---|---|---|
| 10 | O(n!) | перестановки / backtracking |
| 20 | O(2ⁿ) | битовая маска / перебор подмножеств |
| 500 | O(n³) | тройной цикл / DP |
| 5 000 | O(n²) | двойной цикл / DP |
| 1 000 000 | O(n log n) | сортировка, куча, разделяй и властвуй |
| > 1 000 000 | O(n) / O(log n) | хеширование, два указателя, бинарный поиск |

Если сомневаешься — сначала читай ограничения: они намекают на сложность
ожидаемого решения.
